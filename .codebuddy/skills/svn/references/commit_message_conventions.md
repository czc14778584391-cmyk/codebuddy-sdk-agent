# SVN 提交消息规范

## 提交消息格式

遵循常规提交格式:

```
<类型>(<范围>): <主题>

<正文>

<页脚>
```

## 类型说明

- **feat**: 新功能
- **fix**: Bug 修复
- **docs**: 仅文档变更
- **style**: 不影响代码含义的变更(空格、格式化等)
- **refactor**: 既不是修复 bug 也不是添加功能的代码变更
- **perf**: 提升性能的代码变更
- **test**: 添加缺失的测试或修正现有的测试
- **chore**: 构建过程或辅助工具和库的变更
- **revert**: 回退之前的提交

## 示例

### 良好的提交消息

```
feat(auth): 添加使用 JWT 的用户认证

实现了使用 JWT 令牌的 OAuth2 认证流程。
添加了登录/登出接口和用户会话管理。

关闭 #123
```

```
fix(api): 解决用户服务中的空指针异常

修复了用户配置文件缺失时的 NPE 问题。添加了空值检查
和可选字段的默认值。

在问题 #456 中报告
```

```
docs(readme): 更新安装说明

更新了新依赖项的安装指南。
添加了常见问题的故障排除部分。
```

```
refactor(database): 优化查询性能

重写慢查询以使用索引。将平均查询时间
从 500ms 降低到 50ms。
```

### 不好的提交消息

```
修复bug
更新代码
完成
测试提交
```

## 最佳实践

1. **使用祈使语气**: "添加功能" 而不是 "添加了功能" 或 "添加功能中"
2. **主题行保持在 50 个字符以内**
3. **主题行首字母大写**
4. **主题行不要以句号结尾**
5. **正文在 72 个字符处换行**
6. **使用正文解释做什么和为什么,而不是怎么做**
7. **在页脚中引用问题/PR**
8. **将多个逻辑更改分解为单独的提交**

## 常用模式

### 添加新功能
```
feat(模块): 添加新功能描述

简要描述功能的作用及其必要性。

关联到 #123
```

### 修复 Bug
```
fix(模块): 解决特定的 Bug 描述

解释根本原因、如何修复以及考虑的边缘情况。

修复 #456
```

### 更新文档
```
docs(部分): 描述更新的内容

列出添加或修改的具体部分或主题。

更新 v2.0 版本的文档
```

### 性能优化
```
perf(组件): 改善性能指标

描述优化的内容和测得的改进效果。

内存使用减少 30%,加载时间提高 2 倍
```

## 中文提交消息规范

### 类型(中文)
- **feat**: 功能 - 添加新功能
- **fix**: 修复 - 修复 Bug
- **docs**: 文档 - 文档变更
- **style**: 样式 - 代码格式变更
- **refactor**: 重构 - 代码重构
- **perf**: 性能 - 性能优化
- **test**: 测试 - 添加或修改测试
- **chore**: 构建 - 构建工具或辅助变更
- **revert**: 回退 - 回退之前的提交

### 中文提交示例

```
feat(用户): 添加用户注册功能

实现了完整的用户注册流程,包括邮箱验证和密码加密。
添加了注册接口和相关的数据库表。

相关需求: #123
```

```
fix(订单): 修复订单金额计算错误

修复了多商品订单中折扣计算不正确的问题。
修正了折扣应用逻辑,确保正确计算最终金额。

Bug #456
```

```
docs(API): 更新 API 文档

更新了用户模块的 API 接口文档。
添加了新的接口参数说明和示例。

v2.0.0 文档更新
```

### 中英文混合示例

如果团队习惯使用英文提交消息,可以在消息中使用中文描述:

```
fix(user): 修复登录时的验证码问题

Fixed captcha verification issue during login.
Added retry logic for failed captcha validation.

Bug #789
```

## 团队约定

如果你的团队有特定的提交消息规范,请遵循团队的约定。常见的内容包括:

1. **Issue/需求编号**: 在提交消息中引用相关的需求或 Bug 编号
2. **影响范围**: 描述改动影响的模块或功能
3. **测试说明**: 如果需要特殊的测试步骤
4. **兼容性说明**: 如果改动可能影响向后兼容性

## 工具建议

一些工具可以帮助维护提交消息规范:

- **Commitlint**: 自动检查提交消息格式
- **Husky**: Git hooks,在提交前运行检查
- **SVN 钩子 (hooks)**: 在服务器端验证提交消息格式
- **IDE 插件**: 在提交时提供模板和验证

## 注意事项

1. **每次提交应该是一个逻辑完整的单元**,不要将不相关的改动混在一起
2. **提交消息应该清晰、简洁、准确**,让其他开发者能够理解
3. **避免使用 "update"、"fix"、"change" 等过于通用的词**,应该更具体
4. **如果改动较大,应该在提交消息中详细说明**或关联相关的文档/Issue
5. **在提交前仔细审查提交消息**,确保没有拼写错误和表达不清的地方
